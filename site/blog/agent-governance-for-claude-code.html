Agent Governance
for Claude Code

A shared process layer for routing work, freezing plans, and extending repos without leaking local truth into the global layer.

April 16, 2026 12 min read
Scroll

Foreword

Unstructured Claude Code setups fail in the same way. A single CLAUDE.md turns into a junk drawer. Commands stop being entrypoints and start hiding policy. Repo-specific rules leak into the shared install. Planning artifacts drift across chat, scratch files, and commits until nobody can tell which spec actually governed the work.

This system is not a warehouse of agents. It governs authority, routing, lifecycle, extension boundaries, and durable rationale. The right name is agent-governance.

Thesis: the global layer owns process; the repo layer owns local truth. Everything in this RFC follows from that boundary.

The Failure Mode

Claude Code guidance usually collapses into one of two bad states. Either everything gets stuffed into one root memory file, or the docs tree grows without types, boundaries, or ownership rules. Both look fine for a week. Both rot quickly.

The consequences are predictable. Shared process guidance gets mixed with repo-local conventions. Repos either copy-paste the same workflow docs everywhere or invent incompatible local shapes. Commands drift into hidden behavior. Plans lose fidelity as work moves from discussion to implementation. Durable rationale disappears into chat history instead of landing in a record that later contributors can actually find.

The problem is not missing documentation. The problem is missing governance over where process lives, where repo truth lives, and how work moves between the two.

Two Layers, One Rule

The system is deliberately split in two. The global layer in ~/.claude owns the reusable process contract. The active repository owns the local architecture, domain rules, facts, references, and commands that only make sense inside that codebase.

~/.claude/
├── CLAUDE.md                        # Root contract, authority order, phase boundaries
├── commands/                         # Reserved shared agent-* entrypoints
└── docs/agents/                      # Shared process docs
    ├── router/
    ├── policy/
    ├── playbooks/
    ├── reference/
    ├── index/
    └── decisions/

    ├── index/

That split matters because it keeps the framework reusable. Shared docs explain how work should happen. Repo docs explain how this codebase works. The framework never becomes a dumping ground for somebody else's frontend rules, deployment topology, or naming conventions.

The framework owns:
- ~/.claude/CLAUDE.md
- ~/.claude/commands/*.md
- shared workflow, review, verification, and documentation discipline

The repo owns:
- ./CLAUDE.md
- docs/agents/...
- .claude/commands/*.md
- repo-specific architecture, references, facts, and Decision Records

Typed Docs, Not A Blob

The second governance rule is that the docs tree stays typed. Each directory exists for one job. That sounds bureaucratic until you have to extend the system without rereading the whole corpus every time.

Doc Type Job
router/ Decides what to load next once the task is partially understood.
policy/ Defines binding rules that the agent is expected to follow.
playbooks/ Provides ordered workflows such as adoption, verification, and change closeout.
reference/ Holds deeper material once the agent has already narrowed the problem.
index/ Gives stable names to recurring tasks and route lookups.
decisions/ Preserves durable rationale so later contributors do not have to rediscover it.

Commands stay thin for the same reason. A shared command should load canonical docs in the right order. It should not smuggle repo-specific policy into the global layer. That is why the shared namespace stays reserved as agent-*, and repo-local commands are expected to use repo-specific names rather than shadowing framework commands.

Change Bundles: Plans That Survive Context Loss

The most important operational contract in the system is the change bundle. Non-trivial work gets one durable home under docs/changes/NNNNNNN-short-name/. The bundle spans planning, implementation, validation, and closeout, even when the work takes multiple commits.

That matters because the bundle is the handoff boundary between planning and execution, especially in agent-first workflows. The failure is rarely that code is missing. The failure is that the next agent cannot recover why existing code is the way it is, which constraints still matter, and which plan actually governs the change. Without a bundle, that rationale gets split across chat, scratch files, commit messages, and the implementer's memory. With a bundle, the frozen plan, active state, validation evidence, and approved deviations all have one durable home.

docs/changes/0000123-agent-governance/
├── plan.md                         # Exact approved plan, frozen after open
├── state.yaml                      # Presence marks the bundle as active
├── verification.md                 # Validation evidence, kept after closeout
├── review.md                       # Optional, for approved deviations or learnings
└── notes.md                        # Optional scratchpad, removed at close

One detail makes this model much better than a loose planning note: there is no repo-level active index. Open work is discovered from bundle-local state.yaml files. Final numbering is assigned only when the bundle is materialized, not while the agent is still planning.

change: 0000123
slug: agent-governance
status: open
created_at: 2026-04-16T00:00:00Z
updated_at: 2026-04-16T00:00:00Z

It must include enough concrete context that an implementer with no prior repo knowledge can execute the phase without spelunking.

That sentence is the quality bar. A plan is not allowed to hand-wave. If the implementer needs file paths, signatures, schema shapes, migration steps, or acceptance checks, the plan needs to inline them. "Discover during implementation" is a planning failure.

  • /agent-plan-change produces the exact approved plan in chat and stays non-mutating.
  • /agent-open-change rescans docs/changes/, chooses the final number, writes plan.md, state.yaml, and verification.md, then creates the initial bundle-only commit.
  • /agent-close-change removes transient artifacts such as state.yaml and notes.md while leaving the durable record behind.

How Repos Grow Without Polluting The Global Layer

Extension is split into three distinct workflows because adoption, capability growth, and durable rationale are not the same operation. That split is governance, not ornament.

  • /agent-adopt onboards a repo to the framework for the first time.
  • /agent-add-capability adds repo-local policy, playbooks, references, task entries, or repo commands.
  • /agent-record-decision provisions a Decision Record when the repo needs a durable explanation for a lasting boundary.

A concrete example helps. If a repo wants frontend best practices, the framework should define the process for adding that capability, but the actual frontend rules still belong inside the repo.

repo/
├── CLAUDE.md
├── .claude/commands/
│   └── frontend-change.md
└── docs/agents/
    ├── router/
    │   └── capability-map.md
    ├── policy/
    │   └── frontend.md
    ├── reference/
    │   └── frontend-best-practices.md
    └── decisions/
        └── DR-002-frontend-boundaries.md

That is the right split of responsibility. The framework governs how a repo adds the capability. The repo owns the actual content of the capability.

Enterprise Distribution Without Global Sprawl

The enterprise distribution model applies the same boundary at company scale. For one engineer, installing files directly into ~/.claude is fine. At company scale, that path should be treated as the runtime surface, not the authored source of truth.

~/.claude/
├── CLAUDE.md                        # Thin shim to the active installed version
├── commands/                         # Thin agent-* command shims
└── vendor/company-framework/
    ├── current                      # Active version pointer
    ├── 1.4.0/                        # Previous release
    └── 1.5.0/                        # Current release payload

That distinction buys three things immediately: version visibility, fast rollback, and a clean separation between the runtime install and the authored framework source. Repos still own repo-local truth. The shared layer just arrives through a managed install surface instead of manual copying.

  • enterprise policy memory for hard company-wide instructions
  • managed settings for enforced permissions and configuration
  • a user install in ~/.claude for the shared framework runtime
  • repo-local docs and commands for project behavior

For a broad internal rollout, the distribution mechanism should look more like a product than a setup note.

company-claude install
company-claude update
company-claude doctor
company-claude adopt-repo

That installer or manager should install the chosen framework version, flip the active shim target, verify the effective runtime surface, and optionally bootstrap the minimum repo-local adoption files. MCP can augment that with dynamic company services later, but it should not be the primary distribution vehicle for the core process docs.

Tradeoffs

This design is not free. It introduces more files than a single monolithic memory document. Authors need to learn the difference between router, policy, playbooks, references, indexes, and decisions. Non-trivial work now has explicit open and close steps instead of a vague planning note plus some luck.

Those costs are worth paying because they buy auditability, a clean Plan Mode story, and a sustainable extension path. The system scales from one repo to many repos, then from one user install to an enterprise rollout, without changing the core boundary: process stays shared, truth stays local.

What An Implementer Does Next

  1. Install the shared framework in ~/.claude as the process layer.
  2. Adopt the target repo with a minimal local shape: repo CLAUDE.md, repo docs/agents/, and optional repo commands.
  3. Use /agent-plan-change for any non-trivial implementation work.
  4. Materialize the approved plan with /agent-open-change before touching code or evergreen docs.
  5. Keep validation evidence inside the bundle while work proceeds, then run /agent-close-change to remove transient state and preserve the durable record.
  6. Use /agent-add-capability when the repo grows, and /agent-record-decision when the repo needs durable rationale for a lasting choice.

That is the utility of the model. A human can understand it quickly, and an implementer with zero context can execute it without spelunking through old chats or reverse-engineering somebody else's memory layout.
