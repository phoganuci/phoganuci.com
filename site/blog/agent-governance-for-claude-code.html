Framework

A reusable Claude Code framework installed at ~/.claude. The global layer owns process; the repo layer owns local truth.

April 16, 2026 18 min read
Scroll

Summary

This RFC defines a reusable Claude Code framework installed at ~/.claude.

The framework is the shared process layer. It standardizes:

  • the root contract in CLAUDE.md
  • the typed documentation tree for routing, policy, playbooks, references, indexes, and decisions
  • the global agent-* command surface
  • the lifecycle for non-trivial implementation work through change bundles in docs/changes/*
  • the extension model that lets repos add their own local standards without moving repo truth into the global layer

The core rule is simple: the global layer owns process; the repo layer owns local truth.

The appendix contains the complete file skeletons for every file in the framework. An implementer should be able to produce the full working system from this document alone.

Problem

Without a clear structure, Claude Code guidance tends to collapse into one of two bad states:

  • a single oversized CLAUDE.md that mixes routing, rules, rationale, and repo conventions
  • an untyped pile of docs and commands with no clear boundary between shared process and repo-local behavior

That creates predictable failures:

  • process guidance and repo-specific conventions get mixed together
  • commands drift into hidden policy instead of acting as thin entrypoints
  • repos duplicate the same workflow guidance or invent incompatible local structures
  • it becomes unclear where durable rationale should live
  • planning and implementation artifacts accumulate inconsistently across commits

The framework needs to solve those problems without becoming a global dumping ground for frontend, backend, or infrastructure standards that only make sense inside a specific repo.

Goals

  • Provide a reusable process layer that can be installed once in ~/.claude.
  • Make the boundary between the global framework and repo-local extensions explicit.
  • Keep docs typed so routing, policy, playbooks, references, indexes, and decisions each have a clear job.
  • Support Claude Code Plan Mode without requiring writes during planning.
  • Make non-trivial implementation work legible across multiple commits through a consistent bundle model.
  • Keep the global command surface small, namespaced, and process-oriented.
  • Give repos a clean path to add local standards, workflows, and durable rationale.

Non-Goals

  • Define universal frontend, backend, or infrastructure standards in the global framework.
  • Replace repo-local architecture docs with global abstractions.
  • Provide a global command for every domain or team workflow.
  • Maintain a repo-level active-work index for change bundles.
  • Turn the framework into a repo scaffolding engine for arbitrary project structure.

Proposed Design

1. Two-Level Hierarchy

The system has two layers:

  • Global layer in ~/.claude
  • Repo layer inside the active repository

The global layer contains the shared process contract:

~/.claude/
├── CLAUDE.md
├── commands/
│   ├── agent-plan-change.md
│   ├── agent-open-change.md
│   ├── agent-implement-change.md
│   ├── agent-close-change.md
│   ├── agent-adopt.md
│   ├── agent-add-capability.md
│   ├── agent-record-decision.md
│   └── agent-improve.md
└── docs/agents/
    ├── router/
    ├── policy/
    ├── playbooks/
    ├── reference/
    ├── index/
    └── decisions/

Each repo extends that with local docs and optional local commands:

repo/
├── CLAUDE.md
├── .claude/commands/
└── docs/agents/
    ├── router/
    ├── policy/
    ├── playbooks/
    ├── reference/
    ├── index/
    └── decisions/

The global layer explains how work should happen. The repo layer explains how that codebase works.

2. Typed Documentation Tree

The framework uses a typed docs tree so each category has a single job. The invocation order during a typical change is:

Category Job
index/ Loaded first. Maps a work description like "plan change" to its canonical doc (index/workflows.md).
router/ Loaded next. Decides which policy and playbook to load based on the current phase and kind of work (router/phases.md, router/capability-map.md).
policy/ Loaded early, checked throughout. Defines binding rules for the current work (policy/core.md, policy/agent-workflow.md).
playbooks/ Sequences the steps for the current phase. Tells you what to do next (playbooks/change-bundles.md, playbooks/verify.md).
reference/ Loaded on demand when a playbook or policy needs deeper material on a specific topic (reference/review-triggers.md, reference/audit-eng.md).
decisions/ Consulted when questioning a past architectural or workflow choice. Not part of the normal flow (decisions/DR-001-agent-phase-workflow.md).

In short: index names, router selects, policy constrains, playbooks sequence, references deepen, and decisions explain why.

This avoids the common failure mode where one document mixes routing, rules, examples, and rationale without clear boundaries.

3. Root Contract

~/.claude/CLAUDE.md is the root contract. It defines:

  • the meaning of $CLAUDE_HOME
  • the authority order
  • the phase contract
  • the command-surface rules
  • the high-level capability summary

That file is the shared entrypoint Claude Code loads as user memory.

4. Composition Contract

The framework needs a clear answer to "what belongs here versus in the repo?".

docs/agents/router/composition.md provides that contract:

  • the global layer owns shared workflow, verification, review, and documentation discipline
  • the repo layer owns repo-specific architecture, naming, testing rules, references, facts, gotchas, and Decision Records
  • repo commands should use distinct local names instead of relying on shadowing framework commands
  • the framework command surface stays generic and process-oriented

5. Change Bundle Model

Non-trivial implementation work is recorded as a change bundle under docs/changes/NNNNNNN-short-name/.

Canonical shape:

docs/changes/0000123-frontend-boundaries/
├── plan.md
├── state.yaml
├── verification.md
├── review.md      # optional
└── notes.md       # optional, removed at close

Bundle semantics:

  • plan.md is the exact approved plan and freezes once the bundle is opened.
  • state.yaml marks the bundle as active while work is in progress.
  • verification.md captures the validation record and remains after closeout.
  • review.md exists only for approved deviations from the frozen plan or meaningful learnings worth preserving.
  • notes.md is disposable scratch space and is removed during closeout.

One bundle spans one body of work, not one commit. The same bundle can survive across multiple implementation commits until the work is closed.

6. Command Surface and Workflow Lifecycle

Framework commands live under ~/.claude/commands/ and use the reserved agent-* namespace. The surface is intentionally small: eight commands covering the full lifecycle from exploration through implementation to long-term learning.

Explore Phase

Before any non-trivial change, a repo needs to be integrated with the framework. Three commands handle different exploration activities:

/agent-adopt covers first-time repo onboarding: it reads the framework composition contract and produces the repo's local entrypoint, capability map, and initial docs.

/agent-add-capability is the wizard for growing a repo's local layer: adding a new policy, playbook, reference, workflow entry, or command. It inspects what already exists and produces a concrete setup checklist.

/agent-improve is for root-cause doc and routing friction repair. When the framework routes to the wrong doc or a playbook is unclear, this command diagnoses and fixes the underlying issue.

Change Lifecycle: Plan / Open / Implement / Close

Non-trivial implementation work follows a four-phase lifecycle:

/agent-plan-change is the planning entrypoint. It inspects the repo, loads the relevant global and repo-local docs, and produces the exact approved plan content that will become plan.md. It proposes a provisional bundle path and slug but does not write to disk, assign the final number, or commit anything. This makes it safe for Claude Code Plan Mode.

/agent-open-change runs after planning is approved and outside Plan Mode. It rechecks docs/changes/, assigns the final next available number, writes plan.md, state.yaml, and verification.md, and creates the initial bundle commit. That commit contains only the bundle artifacts, not implementation edits.

/agent-implement-change runs after the bundle is open and before making implementation edits. It finds the open bundle, loads the frozen plan and implementation-phase docs, queues verify and review docs for later phases, and establishes the implementation task structure with an explicit checkpoint before commit. The plan is treated as the best understanding at planning time, not as infallible instruction. If the plan conflicts with the actual codebase, the implementation should do the right thing and note deviations for review.md.

/agent-close-change runs after implementation and final verification are complete. It ensures verification.md reflects the final validation evidence, keeps review.md only when justified, removes transient artifacts (state.yaml, notes.md), and leaves the durable record behind.

Open bundles are discovered from docs/changes/*/state.yaml. Final numbering is assigned only by /agent-open-change, which rechecks the filesystem at materialization time. No repo-level active-work index is needed.

Learn Phase

/agent-record-decision is the wizard for provisioning Decision Records. When a repo makes a lasting architectural or workflow choice, this command produces the record so future contributors don't have to rediscover the rationale.

7. Enterprise Distribution Model

For company-scale deployment, ~/.claude should be treated as the runtime installation surface, not the authored source of truth.

The framework should be owned in a versioned internal source repository or package and then installed onto developer machines in a stable vendor path.

Recommended shape:

~/.claude/
├── CLAUDE.md
├── commands/
│   ├── agent-plan-change.md
│   ├── agent-open-change.md
│   ├── agent-implement-change.md
│   ├── agent-close-change.md
│   └── ...
└── vendor/company-framework/
    ├── current
    ├── 1.4.0/
    └── 1.5.0/

In this model:

  • ~/.claude/CLAUDE.md is a thin shim that imports the current installed framework version
  • ~/.claude/commands/ contains thin command shims in the reserved agent-* namespace
  • vendor/company-framework/<version>/ contains the actual framework release
  • repos continue to own repo-local architecture and workflow truth in checked-in docs

This structure gives the organization:

  • controlled rollout of framework versions
  • fast rollback by switching the active version
  • a clear distinction between installed runtime files and authored source
  • a stable global command surface that does not require every repo to vendor a copy

The enterprise rollout should use four layers:

  • enterprise policy memory for hard company-wide instructions
  • enterprise managed settings for enforced permissions and configuration
  • user install at ~/.claude for the shared framework runtime
  • repo-local docs and commands for project-specific behavior

The framework should not rely on manual file copying at scale. It should be shipped through an internal installer or manager, for example:

company-claude install
company-claude update
company-claude doctor
company-claude adopt-repo

That tool should install the chosen framework version, update the active shim target, verify the effective runtime surface, and optionally bootstrap the minimal repo-local adoption files.

The framework may also integrate with MCP for dynamic company services such as service-catalog lookup or policy workflows, but MCP should not be the primary distribution mechanism for the core process docs. The base framework should still arrive through installed memory files and commands so it remains stable, inspectable, and versioned.

8. Repo Capability Growth

/agent-add-capability is the framework wizard for growing an already-adopted repo.

It should:

  • inspect the repo for ./CLAUDE.md, repo docs/agents/..., and .claude/commands/...
  • determine whether the capability needs a policy, playbook, reference, workflow entry, command, or directory CLAUDE.md
  • produce a concrete repo-local checklist with exact file paths and required updates
  • redirect to /agent-adopt if the repo has not been integrated yet

The framework should make the setup path concrete without inventing the repo's standards.

9. Decision Records As The Learning Loop

Repos should use Decision Records to preserve durable rationale over time.

In this system, Decision Records are how a repo learns. Policies say what must be true. Playbooks say how work proceeds. References explain the deeper topic. Decision Records explain why a lasting choice was made.

A repo should record a Decision Record when:

  • a capability setup chooses one approach over meaningful alternatives
  • a new policy depends on non-obvious rationale
  • a workflow introduces a lasting boundary or contract
  • contributors would otherwise have to rediscover the same reasoning repeatedly

A repo should not record a Decision Record for routine feature work or temporary implementation notes.

/agent-record-decision provisions these records deliberately by choosing the next DR-NNN-*.md path and surfacing the other repo docs that should point to that decision.

Example: Frontend Best Practices

If a repo wants frontend best practices, those rules belong in the repo layer, not in the global framework.

A typical repo-local addition would look like this:

repo/
├── CLAUDE.md
├── .claude/commands/
│   └── frontend-change.md
└── docs/agents/
    ├── router/
    │   └── capability-map.md
    ├── policy/
    │   └── frontend.md
    ├── reference/
    │   └── frontend-best-practices.md
    └── decisions/
        └── DR-002-frontend-boundaries.md

In that design:

  • policy/frontend.md contains binding frontend rules
  • reference/frontend-best-practices.md contains deeper frontend guidance
  • router/capability-map.md tells Claude when to load the frontend docs
  • .claude/commands/frontend-change.md is optional ergonomics for repeated repo-local work
  • DR-002-frontend-boundaries.md preserves the durable rationale behind lasting frontend boundaries

The framework owns the process for adding this capability. The repo owns the actual frontend standards.

Tradeoffs

Benefits

  • Shared process is centralized and reusable.
  • Repo-specific standards stay local to the repo.
  • The docs tree is easier to reason about because each type has a clear job.
  • The bundle model makes multi-commit work legible without overloading commits themselves.
  • Plan Mode has a clean non-mutating story.
  • The command surface stays safe because it does not claim generic repo-local names globally.
  • Enterprise rollout can be versioned, audited, and rolled back cleanly.

Costs

  • The system introduces more files than a single monolithic CLAUDE.md.
  • Authors need to understand the difference between routing, policy, playbooks, references, and decisions.
  • Non-trivial work now has explicit open and close steps instead of an informal planning note.
  • Organizations need a lightweight installer or management mechanism instead of relying on manual setup.

These costs are acceptable because they buy a clearer separation of concerns, a better Plan Mode story, and a much cleaner long-term extension path.

Recommendation

If your team uses Claude Code across multiple repositories and you want agents to follow a consistent, auditable process, this framework shape fits that use case:

  • global shared process installed in ~/.claude
  • repo-local architecture and domain guidance kept in the repo
  • typed documentation categories
  • a small namespaced global command surface
  • explicit plan -> open -> implement -> close lifecycle for non-trivial work
  • change bundles under docs/changes/* as the durable work record
  • explicit onboarding, capability-growth, and decision-record workflows
