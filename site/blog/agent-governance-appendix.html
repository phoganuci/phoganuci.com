Root Contract

CLAUDE.md Root contract, authority order, phase boundaries 
# Agent Framework

This file is intended to live at `~/.claude/CLAUDE.md`. It defines the shared process layer for Claude Code: phase workflow, change-bundle lifecycle, verification and review discipline, and the typed routing docs that repos can adopt and extend.

## Path Convention

- `$CLAUDE_HOME` always means `~/.claude`.
- Framework docs live under `$CLAUDE_HOME/docs/agents/...`.
- Repo docs use repo-relative paths such as `./CLAUDE.md`, `docs/agents/...`, and `.claude/commands/...`.
- Do not use `$CLAUDE_HOME` to mean the repo root. Keep the global framework and repo-local docs explicit.

## Read This First

Use the typed docs tree for routing and composition detail:
- `$CLAUDE_HOME/docs/agents/router/phases.md`
- `$CLAUDE_HOME/docs/agents/router/capability-map.md`
- `$CLAUDE_HOME/docs/agents/index/workflows.md`
- `$CLAUDE_HOME/docs/agents/router/composition.md`
- `$CLAUDE_HOME/docs/agents/router/authority-order.md` when you need interpretation notes beyond the precedence guidance below

The typed docs tree is the routing layer. Load only the docs that match the current phase and capability.

## Authority Order

Use `$CLAUDE_HOME/docs/agents/router/composition.md` first to decide whether a rule belongs to the shared framework or the current repo.

When sources still conflict after that split, use this order:
- direct user instructions for the current task
- repo `./CLAUDE.md` and nearest directory `CLAUDE.md` files for repo-specific routing and conventions
- repo-specific `docs/agents/policy/*` for repo architecture, boundaries, naming, and testing rules
- `$CLAUDE_HOME/docs/agents/policy/*` for shared workflow, review, verification, and documentation discipline
- repo-specific `docs/agents/playbooks/*`, then `$CLAUDE_HOME/docs/agents/playbooks/*`
- repo-specific `docs/agents/reference/*`, `facts/*`, `gotchas/*`, `decisions/*`, then `$CLAUDE_HOME/docs/agents/reference/*` and `$CLAUDE_HOME/docs/agents/decisions/*`
- code and tests
- `docs/changes/*`
- external memory or prior-session context

External memory is advisory only. Reload canonical docs at phase boundaries instead of relying on memory.

## Phase Contract

- **Explore**: read this file first, then inspect code, tests, configs, or index docs directly.
- **Plan**: for non-trivial implementation work, use `/agent-plan-change` or an equivalent non-mutating planning flow to produce the exact approved plan that will become `docs/changes/NNNNNNN-short-name/plan.md`.
- **Planning checkpoint**: review the approved plan for completeness and ask the user before opening the change.
- **Open**: before any evergreen-doc or implementation edit, run `/agent-open-change` outside Plan Mode. It materializes the change bundle, writes `plan.md`, `state.yaml`, and `verification.md`, and commits the initial bundle only.
- **Implement**: keep the frozen `plan.md` loaded and limit implementation to the enumerated deltas. One change bundle may span multiple implementation commits. Keep `verification.md` current, and use `notes.md` only as disposable scratch.
- **Replan rule**: if additional repo-tracked edits become necessary, pause and return to planning. Do not rewrite `plan.md`; capture approved deviations or meaningful learnings in the bundle's `review.md`.
- **Implementation checkpoint**: review the implementation and validation, then ask the user before closing the bundle or making the final commit.
- **Cleanup loop**: iterate until the phase is clean. Pause only for blockers, scope changes, or material tradeoffs that need user input.
- **Close**: run `/agent-close-change` at finalization. Remove transient artifacts such as `state.yaml` and `notes.md`, and retain the durable record: `plan.md`, `verification.md`, and `review.md` when it exists.
- **Freeze rule**: after the open-change commit, `plan.md` is immutable. Later deviations or meaningful learnings belong in the bundle's `review.md`, not ad hoc plan edits.
- **Active bundle discovery**: do not maintain a repo-level active index. Open bundles are discovered from `docs/changes/*/state.yaml`, and `/agent-open-change` must recheck numbering on disk before assigning the final path.
- **Verify and commit**: every implementation commit must get normal verification. The open-change commit contains only initial bundle artifacts. The phase-closing commit carries the approved implementation diff and bundle cleanup.

## Capability Summary

- **repo composition / adoption / capability growth**: use `$CLAUDE_HOME/docs/agents/router/composition.md`, then the relevant adoption or extension playbook.
- **domain work**: use the repo's capability map when it has one; otherwise start with the shared capability map.
- **planning / review / doc sweep**: load the relevant policy, playbook, and verification docs as the phase requires.
- **troubleshooting**: load `$CLAUDE_HOME/docs/agents/playbooks/troubleshoot.md`, then escalate to repo reference docs.

## Command Surface

- Framework commands installed under `~/.claude/commands/` must use the reserved `agent-*` namespace.
- Repo commands installed under `.claude/commands/` should use distinct repo-local names.
- Keep framework commands process-oriented. Repo-specific scaffolding belongs in repo-local commands.

Commands

commands/agent-plan-change.md Non-mutating implementation plan 
---
description: Prepare a non-mutating implementation plan for a change bundle.
---

Use this command for non-trivial implementation work when you need an approved plan before any repo writes.

Workflow:
1. Inspect the repo and load these docs in order:
   - `$CLAUDE_HOME/CLAUDE.md`
   - `$CLAUDE_HOME/docs/agents/router/phases.md`
   - `$CLAUDE_HOME/docs/agents/policy/agent-workflow.md`
   - `$CLAUDE_HOME/docs/agents/playbooks/change-bundles.md`
   - `$CLAUDE_HOME/docs/agents/playbooks/agent-phase-workflow.md`
   - repo-local policy and playbook docs required for the change
2. Produce the exact approved plan body that will become `docs/changes/NNNNNNN-short-name/plan.md`.
3. Propose a provisional bundle path and slug, but do not write files or assign a final number.
4. Review the plan with the user before any implementation work starts.
5. Instruct the next execution step to run `/agent-open-change` outside Plan Mode.

Rules:
- Safe in Plan Mode because it is non-mutating.
- Do not create `docs/changes/*`, edit code, edit evergreen docs, or commit anything.
- Do not compress or rewrite the approved plan when handing it off.

Pass any `$ARGUMENTS` through to the planning flow.
commands/agent-open-change.md Materialize bundle, commit 
---
description: Materialize an approved change bundle and commit the initial bundle only.
---

Use this command after planning is approved and outside Plan Mode.

Workflow:
1. Confirm that an approved plan already exists in chat or other approved task context.
2. Load relevant docs in order.
3. Re-scan `docs/changes/` and choose the final next available change number.
4. Create the bundle directory and write `plan.md`, `state.yaml`, `verification.md`.
5. Commit the initial bundle. That commit must contain only the bundle artifacts.
6. Stop after the bundle commit. Do not make implementation edits.

Rules:
- This command is the only step that materializes the initial bundle.
- If the approved work already has an open bundle, resume it.
- Do not edit code or evergreen docs here.

Pass any `$ARGUMENTS` through to the bundle-opening flow.
commands/agent-implement-change.md Load implementation phase 
---
description: Load the implementation phase for an open change bundle.
---

Use this command after `/agent-open-change` has committed the initial bundle and before making implementation edits.

Workflow:
1. Find the open bundle by scanning `docs/changes/*/state.yaml`.
2. Load the frozen `plan.md` from that bundle.
3. Load these docs in order:
   - `$CLAUDE_HOME/CLAUDE.md`
   - `$CLAUDE_HOME/docs/agents/policy/agent-workflow.md`
   - `$CLAUDE_HOME/docs/agents/playbooks/agent-phase-workflow.md`
   - `$CLAUDE_HOME/docs/agents/playbooks/change-bundles.md`
   - repo-local domain policy and change playbook for the work area
4. Queue these docs for later phases:
   - `$CLAUDE_HOME/docs/agents/playbooks/verify.md` at verify
   - `$CLAUDE_HOME/docs/agents/playbooks/review.md` at commit
   - `$CLAUDE_HOME/docs/agents/playbooks/docs-workflow.md` at commit
5. Set up the implementation task structure:
   - Load implementation docs
   - Domain-specific implementation tasks from the plan
   - Verify: run baseline checks, then domain-specific verification
   - Implementation checkpoint: review against plan, check for shortcuts, ask the user
   - Commit: load review docs, write `review.md` if deviations exist, commit

Rules:
- The plan is the best understanding at planning time. It is not infallible.
- If the plan specifies something that does not match the actual codebase, fix it rather than blindly following.
- Look for opportunities to make the code cleaner and the architecture more consistent with existing patterns.
- If a plan detail is wrong or incomplete, implement the right thing and note the deviation for `review.md`.
- No shortcuts or hacks. "The plan is fallible" is not license to skip work. If doing the right thing requires effort the plan did not anticipate, do the effort.
- Ask the user before finalizing.

Pass any `$ARGUMENTS` through to the implementation flow.

---
description: Close an open change bundle and clean up transient bundle artifacts.

Use this command after implementation and final verification are complete.

- Retain `plan.md`, `verification.md`, and bundle `review.md` when it exists.

# Composition Contract

The framework owns: `~/.claude/CLAUDE.md`, `~/.claude/commands/*.md`, shared policy, playbooks, reference, and decisions.

The repo owns: `./CLAUDE.md`, optional directory entrypoints, repo `docs/agents/...`, repo `.claude/commands/*.md`.

## Composition Rules
1. `$CLAUDE_HOME` always means `~/.claude`.
2. Repo docs use repo-relative paths.
3. Repo `./CLAUDE.md` points to framework docs it relies on, then adds repo-specific routing.
4. Repos extend with their own capability-map and workflow index.
5. Repo-specific docs own repo behavior. Framework docs own shared workflow.
6. Repo commands use distinct names, not same-name shadowing.
7. Commands stay process-oriented.

## Recommended Adoption Shape
Start small: `./CLAUDE.md`, one policy, one playbook. Add more only when needed.

# Core Policy

## Non-negotiables
- Proactively flag improvements when producing weak results.
- Tests must test behavior, not merely prove code does not panic.
- New code ships with tests in the same commit.
- Fix pre-existing gaps in the touched area.
- Extract duplicated patterns when the shared abstraction is clear.
- Ask instead of guessing on unclear architecture choices.
- Research 3-5 best-in-class implementations for user-facing features.
- Main is evergreen. Do not leave broken tests, stale docs, or weak implementations.
- Fix holistically, not just the first symptom.
- Use repo-supported mechanisms over workarounds.
- Do not treat prior plans as scripture.
- Do not use bare literals for domain concepts.

## Verification and Review
- Run verification before committing.
- Run engineering review for phase completion, not every intermediate commit.
- Missing a phase artifact or clean review is a workflow failure.
- Show all findings to the user.
- Honest self-check for shortcuts before finalizing.

## Documentation
- Follow the repo-wide phase artifact contract.
- Evergreen doc edits follow the same workflow as code changes.
- Update docs in the same commit when behavior changes.

# Agent Workflow Policy

## General Rules
- All phase artifacts go to the change-bundle directory, never /tmp.
- No repo-level active index. Open work discovered from bundle-local state.yaml.
- Do not silently filter review findings.
- Ask the user before moving past either checkpoint.

## Written Spec Rules
- The governing spec is on-disk plan.md inside docs/changes/NNNNNNN-short-name/.
- Must be verbatim, lossless copy of the approved plan.
- Must enumerate every planned delta.
- Must include enough context for a zero-context implementer.
- Missing deltas or "discover during implementation" are planning failures.

## Phase Checkpoint Rules
- Planning checkpoint verifies "Sufficient implementation context" and "Zero-context implementable".
- Implementation checkpoint verifies "Plan deltas applied" and "Validation coverage".
- Skipping the review or user prompt at either checkpoint is a workflow failure.
- After open-change commit, the written spec is frozen.

# Change Bundles Playbook

# Agent Phase Workflow

## Planning
1. Decide if work needs a change bundle.
2. Use /agent-plan-change in Plan Mode.
3. Draft lossless, self-contained implementation contract.
4. Review plan and ask user before starting.

1. Finalize verification.md.

3. Run /agent-close-change.

# DR-001 Agent Phase Workflow

The repo adopts a policy-first agent phase workflow for implementation work.

Rationale:
- CLAUDE.md and the typed docs tree are the shared entrypoint for Claude.
- Strong repo policy is easier to inspect, maintain, and teach in one shared docs tree.
- The agent reviews its own work at each checkpoint and asks the user before moving past it.

Consequence:
- Workflow is strongly guided and auditable.
- Users can still violate policy by bypassing the workflow, but that violation is explicit.
